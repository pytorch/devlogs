## What mergedog is

[mergedog](https://github.com/ezyang/mergedog) is a small Python harness that takes one approved
`pytorch/pytorch` PR and shepherds it through CI to the point a
human can comment `@pytorchbot merge`. One process per PR,
foreground, synchronous. It does not replace pytorchbot, doesn't
type the merge command itself, and only operates on PRs the
maintainer has already approved.

The four pillars that govern every design decision, in priority
order:

- **Adoption.** Once we acknowledge a PR as good, we take full
  control. The original author is not involved in the rest of the
  process.
- **Security.** It's actively harmful, not just inconvenient, if
  the contributor pushes after approval — they could introduce code
  the maintainer never reviewed. Default to security; explain any
  tradeoff.
- **Autonomous.** Ask for a PR to be merged, then don't touch it
  until it's done. Prefer HALT-with-message over interactive
  prompts.
- **Synchronous.** One foreground process per PR, even when the
  mux runs many in parallel.

A fifth, complementary preference: **smallest implementation**.
mergedog is a personal harness for one user — no config knobs,
plugin layers, or generic interfaces for hypothetical needs.
Security goals override this default; nothing else does.

The shepherd's loop:

1. Approve any `action_required` workflow runs (external-author PRs
   start gated until a maintainer presses "Approve and run").
2. Read CI status by enumerating all workflow runs for the PR head.
3. **Failed** → invoke Claude in "fix-CI" mode. Claude either makes
   one `[MERGEDOG]`-prefixed commit fixing the failures, or makes no
   commit because the failures look spurious.
4. **Passed** → if the merge-base with `main` is older than a week,
   merge `main` in (handing conflicts to Claude). Then apply
   `ciflow/trunk` and keep polling.
5. **Trunk green** → post a "mergedog handoff" comment summarizing
   every Claude session and wait for `@pytorchbot merge`.

## Security model

The maintainer's "approve" attests to one thing: the diff visible in
the GitHub UI at that SHA is okay to land. mergedog treats anything
outside that boundary as untrusted.

- **Trusted commits** are either (a) approved by the maintainer at
  that exact SHA, or (b) `[MERGEDOG]`-prefixed and produced by
  mergedog itself. Every poll re-fetches the PR head and halts if
  it's moved to anything else — a contributor pushing after approval
  is treated as actively harmful, not just inconvenient.
- **What the agent sees** is a single squashed commit at the PR's
  tip. Its prompt forbids `git log`, `git show`, `git blame`, and
  reading under `.git/logs/`. PR title, body, and comments come in
  through a sidecar file that's pre-sanitized by `mergedog/sanitize.py`
  (HTML comments stripped, zero-width / bidi-override Unicode
  removed, `<details>` tags unwrapped).
- **Separation of duties.** The agent is forbidden from `git push`
  and from any `gh` command that mutates the PR. It makes a local
  commit; the harness validates the subject, records the SHA in the
  trust DB, and pushes.

## Using it

```
$ python -m mergedog 173321
```

PR URL also works. Useful flags:

- `--max-base-age DAYS` (default 7) — when to merge `main` in first.
- `--accept-divergence` — proceed even if the head moved past the
  approval (only after re-reviewing the new commits).
- `--ignore-sev` — don't park on open `ci: sev` issues.

State lives at `~/.mergedog/`: per-PR logs in `logs/<pr>.log`, trust
state in `state/<pr>.json`, worktree in `worktrees/<pr>/`. Ctrl-C
anywhere; restart with the same command and it resumes.

A `mergedog.mux` TUI runs many shepherds in parallel:

```
$ python -m mergedog.mux
```
